# AI Rail Three Interaction Models

## Prompt Sources

- Coding agent prompt: `rail next --copy`
- Reviewer prompt: `rail verify --copy`
- Planning prompt: `rail plan --copy`
- Phase-audit prompt: `rail phase --copy`
- New-chat handoff: `rail handoff --for chatgpt --copy`
- Tool-specific handoff: `rail handoff --for codex --copy` or `rail handoff --for claude --copy`
- Export root AI instruction files: `rail export`

Do not manually invent prompts when AI Rail can generate them.

`.rail/PROJECT.md` is local project memory, roadmap, phase tracker, next-task direction, and local mirror for AI handoffs. GitHub Issues are the task execution layer. The GitHub roadmap issue is the remote roadmap mirror. Coding happens one issue at a time; phase audit happens after several shipped issues and is not coding.

## Model 1: Codex-Based

An AI reviewer such as ChatGPT, Claude, or another LLM plans/reviews. Codex, Claude Code, Cursor, Aider, or another coding agent edits locally. You validate and ship through AI Rail.

Use for larger or multi-file work.

Normal workflow:

```text
Create GitHub issue -> rail next --copy -> paste prompt into Codex/Claude/Cursor/Aider -> agent edits only issue scope -> rail verify --copy -> AI reviewer audit -> rail ship
```

For Codex, Claude Code, Cursor, or Aider, use the prompt generated by `rail next --copy`. The AI reviewer plans/reviews; the coding agent edits locally; the AI reviewer audits after `rail verify --copy`.

The coding agent must stay on the active issue, touch only necessary files, not commit, not close issues, and not run broad/full test suites unless the issue explicitly asks for them. Run only focused checks related to the issue, or the exact checks requested by the user or AI Rail.

## Model 2: Patch-Based

An AI reviewer writes a `.patch`. You apply it locally with AI Rail.

Use for small, surgical edits.

Workflow:

```text
Create or choose issue -> ask an AI reviewer for a surgical patch -> apply patch locally -> rail verify --copy -> AI reviewer audit -> rail ship
```

Patch-producing AI must produce a minimal patch, touch only issue-required files, avoid broad refactors, include apply commands, include rollback commands when possible, and suggest focused checks. It must not claim tests were run unless the user actually ran them.

## Model 3: AI-Direct / GitHub-Direct

An AI reviewer or another AI tool changes GitHub directly.

Use only for low-risk docs, config, or small static changes unless risk is explicitly accepted.

AI-direct must state it is using Model 3, change only requested GitHub files, avoid broad code changes, and never pretend the local repo is updated before the user pulls.

Mandatory local sync commands:

```powershell
git fetch origin
git pull --ff-only
git status
```

If AI creates a branch or PR, it must provide exact checkout/fetch commands for that branch or PR. After pulling, run `rail verify --copy` locally.

## Review, Planning, And Phase Audit

- Review/audit: identify drift, check changed files against issue scope, check focused checks, decide whether ship is safe, and provide exact next terminal commands.
- Issue planning: provide a GitHub issue title, issue body, issue creation command when possible, and the next `rail` command.
- Phase audit: compare completed issues against project direction, identify missing/fake/incomplete work, propose next issues, and do not implement unless explicitly asked.
